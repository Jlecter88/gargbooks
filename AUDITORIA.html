Gerado em 09/07/2026 · Projeto: gargbooks (Next.js 16 + Firebase + TypeScript)

Resumo Executivo

O Gargbooks é uma plataforma literária premium com curadoria gótico/premium. O projeto já possui toda a infraestrutura necessária para ter um acervo massivo, mas as obras não estão sendo carregadas no site por 4 problemas centrais:

  1. O arquivo livros-mock.json só contém 7 dos 70+ livros que o orquestrador já baixou e traduziu.
  2. O orquestrador (scripts/orchestrator.js) depende de LM Studio + Stable Diffusion locais que não estão rodando, travando o pipeline.
  3. Não há integração com APIs brasileiras (Domínio Público, Brasiliana, Open Library) para acervo nacional.
  4. O Firebase Firestore está configurado mas não populado — toda a persistência é localStorage.

4. Plano de Execução — Ordem Recomendada

As etapas estão ordenadas por dependência: cada fase desbloqueia a próxima.

Fase 1 — Resgate Imediato do Acervo (dias 1-3)

Sem depender de LM Studio ou Firebase. Usar apenas Node.js + scripts existentes.

1
Criar endpoint /api/dominio-publico (análogo ao /api/gutenberg)

Criar src/app/api/dominio-publico/route.ts que baixa do dominiopublico.gov.br/download/texto/{id}.txt, limpa headers/footers, e retorna texto puro com cache. Usar o mesmo padrão do gutenberg/route.ts.

// GET /api/dominio-publico?id=bn000067 // Baixa de http://www.dominiopublico.gov.br/download/texto/bn000067.txt // Limpa cabeçalho/rodapé do governo // Retorna text/plain com Cache-Control: public, max-age=86400
Crítico   ~1h de trabalho
2
Adicionar TODAS as 50+ obras brasileiras ao livros-mock.json

Usar as listas do orquestrador (DOMINIO_PUBLICO_WORKS, linhas 186-246) para gerar entries completas com downloadFile apontando para o novo endpoint, traduções (pt-br default), capas via Open Library covers, e gradientes.

Crítico   ~4h de trabalho
3
Criar script de inicialização (bootstrap) que baixa as obras brasileiras

Script Node.js que baixa TODAS as 50+ obras do Domínio Público de uma só vez, gera entries para o livros-mock.json, e prepara tudo para o deploy. Independente de LM Studio.

node scripts/bootstrap-brasil.js // 1. Baixa 50+ obras Domínio Público → /downloads // 2. Gera entries para livros-mock.json // 3. Salva progresso
Alto impacto   ~3h de trabalho

Fase 2 — APIs e Frontend (dias 4-6)

5
Criar obra brasileira em destaque com aba autônoma "Acervo Brasileiro"

Nova ContentTab "brasil" no frontend que filtra publicDomain: true, language: "pt-br". Adicionar filtro por autor (Machado, Alencar, etc.) e período literário (Romantismo, Realismo, etc.).

Médio   ~3h de trabalho
6
Configurar Google Books API Key

Criar chave no Google Cloud Console, ativar Books API, adicionar ao .env.local como NEXT_PUBLIC_GOOGLE_BOOKS_KEY, atualizar /api/amazon/route.ts para usar a key. Isso libera 1000 req/dia.

// .env.local GOOGLE_BOOKS_API_KEY=sua_chave_aqui
Alto impacto   ~30min
7
Resolver chunks no leitor de livros

O public/downloads/ tem arquivos chunkados (.chunk.0.txt). O leitor (src/app/livros/[id]/page.tsx) precisa detectar chunks, baixá-los em paralelo, concatená-los na ordem correta, e exibir o texto completo.

async function loadFullText(downloadFile: string) { const base = downloadFile.replace('.txt', ''); const chunks = []; for (let i = 0; ; i++) { const res = await fetch(`${base}.chunk.${i}.txt`); if (!res.ok) break; chunks.push(await res.text()); } return chunks.length > 0 ? chunks.join('\n') : await fetch(downloadFile).then(r => r.text()); }
Alto   ~2h de trabalho
8
Adicionar carregamento lazy + paginação na grade

Com 70+ livros, a grade atual (renderiza todos de uma vez) vai ficar lenta. Implementar IntersectionObserver com carregamento de 12-16 por página, ou paginação clássica.

Baixo   ~2h de trabalho

Fase 3 — Firebase e Persistência (dias 7-10)

9
Rodar db-feeder.js para popular Firestore

O script scripts/db-feeder.js foi criado para migrar os dados do JSON para o Firestore. Executá-lo vai popular a base e permitir que o site leia do Firestore em vez de localStorage.

node scripts/db-feeder.js // Usa firebase-admin para: // - Upload de livros → firestore collection "books" // - Upload de contos → firestore collection "contos" // - Upload de covers → storage bucket
Alto impacto   ~2h de trabalho
10
Migrar BookContext para ler do Firestore (com fallback para JSON)

Modificar BookContext.tsx para tentar carregar do Firestore primeiro, e cair para o JSON local se falhar. Isso permite escalar para milhares de livros sem engessar o bundle.

Médio   ~4h de trabalho
11
Mover textos completos para Firebase Storage

Os TXT em public/downloads/ ficam no bundle. Mover para Firebase Storage, com o leitor baixando sob demanda via URL assinada ou pública. Reduz o tamanho do deploy em 95%.

Médio   ~3h de trabalho

Fase 4 — Orquestrador Autônomo (dias 11-15)

12
Modificar orquestrador para modo "offline" (sem LM Studio)

Separar o pipeline em 2 modos: (a) download-only: baixa textos do Gutenberg/DP, gera metadados, salva no Firestore — funciona sem LM Studio. (b) full: com IA para tradução e contos. O modo (a) já desbloqueia todo o acervo.

Alto impacto   ~4h de trabalho
13
Auto-descoberta de novas obras do Gutenberg

O orquestrador já tem lógica de auto-discovery (linhas ~1000+). Usar Gutendex para descobrir novas obras populares em português e adicionar automaticamente ao acervo.

Baixo   ~2h de trabalho

Fase 5 — Contínuo (dias 16+)

14
Depurar truncamento de textos grandes no leitor

Alguns textos no livros-mock.json estão truncados (especialmente os que têm fullText inline em vez de downloadFile). Garantir que todos os livros usem downloadFile + chunk loading.

Médio   ~2h de trabalho
15
Implementar Amazon PAAPI para preços reais

Substituir os preços estimados por preços reais da Amazon via Product Advertising API v5. Requer conta de afiliado e configuração de Signed Requests.

Baixo   ~4h de trabalho
16
Adicionar Open Library como fallback de capas e metadados

Quando Google Books rate-limitar, usar Open Library para buscar metadata e capas. Implementar rota /api/openlibrary.

Baixo   ~2h de trabalho

Se não, baixar o arquivo único normalmente 4. Tratar erros: se chunks falharem, tentar fallback para fullText do JSON 5. Adicionar indicador de progresso "Carregando texto completo... (X%)" 6. Usar Promise.all para baixar chunks em paralelo quando possível A função deve ficar em src/context/BookContext.tsx ou em um utils/textLoader.ts Incluir tipagem TypeScript completa. Verificar com npm run build.
